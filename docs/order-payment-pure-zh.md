# 电商订单支付系统：USL 纯中文阅读版

这是一份纯中文版本，用来帮助理解“电商订单支付系统”应该如何被 USL 这种语义源语言表达。它不是给解析器读取的版本，而是给人阅读的版本。

---

# 一、系统目标

这个系统的目标是：

让买家可以为待支付订单完成付款，并且保证下面几件事始终一致：

- 订单状态正确
- 支付状态正确
- 库存预占正确
- 支付金额不能被篡改
- 支付回调可以重复到达但不会重复处理
- 订单支付成功事件不能重复发出
- 每一次支付回调都必须有审计日志

一句话说：

> 这个系统要解决的不是“怎么调用支付接口”，而是“如何保证订单、支付、库存、事件和审计在各种异常情况下仍然正确”。

---

# 二、参与者

## 买家

买家是已经注册的用户。

买家可以做这些事情：

- 创建订单
- 点击支付
- 在支付成功前取消订单
- 查看订单支付结果

## 第三方支付服务

第三方支付服务负责处理真实支付。

它会向系统发送支付回调，例如：

- 支付成功回调
- 支付失败回调
- 重复支付回调

系统不能无条件相信第三方回调，必须验证签名和金额。

---

# 三、核心业务对象

## 订单

订单代表买家购买商品后形成的交易记录。

订单包含：

- 订单 ID
- 买家 ID
- 订单状态
- 订单总金额
- 币种

订单可能有这些状态：

- 待支付
- 已支付
- 支付失败
- 已取消
- 已退款

订单状态不能随意变化。

允许的变化是：

- 待支付 → 已支付
- 待支付 → 支付失败
- 待支付 → 已取消
- 已支付 → 已退款

不允许的变化包括：

- 已取消 → 已支付
- 已退款 → 已支付
- 支付失败 → 已支付，除非重新创建新的支付流程

## 支付单

支付单代表一次支付尝试。

支付单包含：

- 支付单 ID
- 订单 ID
- 第三方支付交易号
- 支付状态
- 支付金额
- 币种

支付单可能有这些状态：

- 已创建
- 支付处理中
- 支付成功
- 支付失败
- 已取消

允许的变化是：

- 已创建 → 支付处理中
- 支付处理中 → 支付成功
- 支付处理中 → 支付失败
- 已创建 → 已取消

## 库存预占

库存预占用于防止超卖。

在买家进入支付前，系统应该先预占库存。

库存预占可能有这些状态：

- 已预占
- 已释放
- 已消耗

允许的变化是：

- 已预占 → 已消耗
- 已预占 → 已释放

支付成功时，库存预占变成已消耗。

支付失败或订单取消时，库存预占变成已释放。

## 订单支付成功事件

订单支付成功事件表示订单已经完成支付。

它通常会触发后续动作，例如：

- 通知履约系统
- 发送站内消息
- 生成财务流水
- 更新用户订单列表

这个事件必须只发出一次。

即使支付成功回调重复到达，也不能重复发出订单支付成功事件。

## 支付审计日志

支付审计日志记录每一次支付回调的处理情况。

它应该记录：

- 支付单 ID
- 回调动作
- 回调内容的哈希值
- 创建时间

审计日志的意义是：

- 方便排查支付问题
- 方便追踪重复回调
- 方便审计资金相关操作
- 方便证明系统没有重复处理

# 四、主要业务流程

## 1. 创建支付

当买家点击支付订单时，系统应该先检查：

- 订单是否处于待支付状态
- 库存是否已经预占

如果条件满足，系统应该：

- 创建支付单
- 使用订单的服务端金额创建第三方支付请求
- 将支付单状态标记为支付处理中

如果条件不满足，系统应该拒绝支付，并返回：

```text
订单当前不可支付
```

这里最重要的规则是：

> 支付金额必须来自服务端订单金额，不能相信前端传入的金额。

## 2. 处理支付成功回调

当第三方支付服务发送支付成功回调时，系统必须先检查：

- 回调签名是否有效
- 支付金额是否等于订单总金额
- 订单是否仍然处于待支付状态

如果全部满足，系统应该：

- 将支付单标记为支付成功
- 将订单标记为已支付
- 将库存预占标记为已消耗
- 发出订单支付成功事件
- 记录支付审计日志

如果任何条件不满足，系统应该拒绝回调，并返回：

```text
支付回调无效
```

这里最重要的规则是：

> 支付成功不是由前端按钮决定的，而是由可信的支付回调、签名验证、金额校验和订单状态共同决定的。

## 3. 处理支付失败回调

当第三方支付服务发送支付失败回调时，系统必须检查：

- 回调签名是否有效
- 订单是否仍然处于待支付状态

如果条件满足，系统应该：

- 将支付单标记为支付失败
- 将订单标记为支付失败
- 释放库存预占
- 记录支付审计日志

如果条件不满足，系统应该拒绝回调，并返回：

```text
支付回调无效
```

这里最重要的规则是：

> 支付失败后，库存不能继续被占用，否则会造成库存被无意义冻结。

## 4. 支付成功前取消订单

当买家在支付成功前取消订单时，系统必须检查：

- 订单是否仍然处于待支付状态

如果满足，系统应该：

- 将订单标记为已取消
- 将支付单标记为已取消
- 释放库存预占

如果订单已经支付或退款，系统应该拒绝取消，并返回：

```text
订单当前不可取消
```

这里最重要的规则是：

> 订单一旦支付成功，就不能再按“未支付取消”的流程处理。

## 5. 处理重复支付回调

第三方支付服务可能会多次发送同一个支付成功回调。

当重复支付成功回调到达时，如果订单已经是已支付状态，系统应该：

- 保持订单为已支付
- 保持支付单为支付成功
- 不重复发出订单支付成功事件
- 记录支付审计日志

系统可以返回重复回调提示，但不能造成重复副作用。

这里最重要的规则是：

> 支付回调必须幂等。

幂等的意思是：

同一个回调处理一次和处理多次，最终结果应该一样。

---

# 五、必须遵守的业务规则

这个系统必须遵守下面这些规则：

1. 必须验证支付回调签名。
2. 必须保证重复支付回调是幂等的。
3. 不能重复发出订单支付成功事件。
4. 不能相信前端传入的支付金额。
5. 必须使用服务端订单金额作为支付金额。
6. 支付前必须先预占库存。
7. 支付失败或订单取消时必须释放库存。
8. 每一次支付回调都必须记录审计日志。

这些规则比具体代码更重要。

因为不同语言、不同框架、不同数据库都可以实现这些规则，但系统是否正确，取决于这些规则有没有被稳定满足。

# 六、错误情况

## 订单当前不可支付

出现条件：

- 订单不是待支付状态
- 库存没有预占
- 订单已经取消
- 订单已经支付

用户提示：

```text
订单当前不可支付
```

## 支付回调无效

出现条件：

- 回调签名无效
- 回调金额和订单金额不一致
- 回调对应的支付单不存在
- 回调对应的订单状态不允许继续处理

用户或系统提示：

```text
支付回调无效
```

## 订单当前不可取消

出现条件：

- 订单已经支付
- 订单已经退款
- 订单不处于待支付状态

用户提示：

```text
订单当前不可取消
```

## 重复支付回调

出现条件：

- 同一个支付成功回调已经处理过
- 订单已经处于已支付状态
- 支付单已经处于支付成功状态

系统提示：

```text
重复支付回调
```

# 七、验收测试

## 测试一：创建支付必须要求库存已预占

前置条件：

- 订单处于待支付状态
- 库存已经预占

动作：

- 买家点击支付订单

预期结果：

- 支付单被创建
- 支付单进入支付处理中
- 库存在支付前已经被预占

## 测试二：支付成功回调可以正确完成订单

前置条件：

- 订单处于待支付状态
- 支付单处于支付处理中
- 库存处于已预占状态

动作：

- 第三方支付服务发送有效支付成功回调

预期结果：

- 系统验证支付回调签名
- 支付单变成支付成功
- 订单变成已支付
- 库存预占变成已消耗
- 订单支付成功事件被发出
- 支付审计日志被记录

## 测试三：伪造支付回调会被拒绝

前置条件：

- 支付回调签名无效

动作：

- 第三方支付服务发送回调

预期结果：

- 系统返回支付回调无效
- 订单保持待支付状态

## 测试四：金额篡改会被拒绝

前置条件：

- 前端传入金额和订单服务端金额不一致

动作：

- 第三方支付服务发送支付成功回调

预期结果：

- 系统返回支付回调无效
- 系统使用服务端订单金额
- 系统不信任前端传入金额

## 测试五：重复支付回调必须幂等

前置条件：

- 订单已经处于已支付状态
- 支付单已经处于支付成功状态

动作：

- 第三方支付服务再次发送同一个支付成功回调

预期结果：

- 处理结果保持一致
- 订单仍然是已支付
- 支付单仍然是支付成功
- 不重复发出订单支付成功事件
- 仍然记录支付审计日志

## 测试六：取消订单后必须释放库存

前置条件：

- 订单处于待支付状态
- 库存处于已预占状态

动作：

- 买家取消订单

预期结果：

- 订单变成已取消
- 库存预占变成已释放

# 八、AI Agent 修改边界

如果把这份语义交给 AI Agent 实现，允许它修改：

- 订单模块
- 支付模块
- 库存模块
- 订单支付相关测试

不允许它修改：

- 登录认证模块
- 历史数据库迁移
- 与本次支付功能无关的旧系统逻辑

这样可以防止 AI 为了实现支付功能而误改不相关模块。

---

# 九、完成标准

这个系统只有在满足以下条件时，才算完成：

- 所有支付相关测试通过
- 支付回调幂等性已验证
- 支付回调签名验证已文档化
- 每一次支付回调都有审计日志
- 不可能重复发出订单支付成功事件
- 不破坏现有订单 API 行为

# 十、这个案例体现的 USL 思想

这个案例说明：

代码不是第一层。

第一层应该是：

- 对象
- 状态
- 流程
- 规则
- 错误
- 测试
- 边界
- 完成标准

当这些语义被清楚表达出来以后，Python、Java、Go、C++ 或 TypeScript 都只是实现目标。

USL 真正想解决的问题是：

> 如何把人的业务意图变成 AI 能执行、测试能验证、工程边界能约束的软件语义源。
